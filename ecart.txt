===========================================================================
PARTIE A — ECARTS CDC vs REALISE
===========================================================================

-----------------------------------------------------------------------
A1. SOCKET.IO → WEBSOCKET NATIF FASTAPI  [écart intentionnel]
-----------------------------------------------------------------------
CDC prévu   : Communication temps réel via Socket.IO
Réalisé     : WebSocket natif FastAPI (Starlette), sérialisation orjson bytes
Raison      : - FastAPI intègre nativement WebSocket sans dépendance externe
              - Socket.IO ajoute ~80 Ko au bundle frontend + ~10-30 % overhead
              - orjson 3-5x plus rapide que JSON standard pour les frames fréquentes
              - Reconnexion automatique réimplémentée dans WSService (backoff expo)
Impact      : Aucun impact fonctionnel. WebSocket natif équivaut à Socket.IO
              pour ce cas d'usage.

-----------------------------------------------------------------------
A2. GRILLE 20x20 → 25x25  [écart intentionnel]
-----------------------------------------------------------------------
CDC prévu   : Grille de jeu 20x20
Réalisé     : Grille de jeu 25x25 (GRID_SIZE=25 dans config.py, variable d'env)
Raison      : Plus d'espace pour les agents IA, parties plus longues et
              visuellement intéressantes. Les algorithmes sont agnostiques à la taille.
Impact      : Score moyen A* plus élevé qu'attendu sur une 20x20.

-----------------------------------------------------------------------
A3. DQN (réseau de neurones) NON REALISE  [écart par priorité MoSCoW : Could]
-----------------------------------------------------------------------
CDC prévu   : Possibilité d'implémenter un Deep Q-Network (DQN)
Réalisé     : Q-Learning tabulaire uniquement (Q-table JSON, 254 états visités)
Raison      : DQN nécessite PyTorch + GPU recommandé. Complexité très élevée
              hors scope d'une SAE. La Q-table tabulaire est suffisante pour
              la démonstration et reste explicable (O(1), 11 features lisibles).
Impact      : Score Q-Learning plafonné (~23 max, ~9 moyen sur 80 épisodes).
              DQN permettrait d'atteindre 50-100+ avec assez d'entraînement.

-----------------------------------------------------------------------
A4. REPLAY FRAME PAR FRAME NON FONCTIONNEL  [écart partiel]
-----------------------------------------------------------------------
CDC prévu   : Replay des parties enregistrées frame par frame
Réalisé     : - La route backend GET /api/stats/replay/{game_id} EXISTE et
                retourne les événements de la partie (table game_events).
              - La table game_events enregistre chaque mouvement (move, food,
                collision) avec timestamp, direction, score.
              - Le composant ReplayViewer.jsx existe dans le code source.
              - MAIS : ReplayViewer.jsx n'est pas connecté à la route backend.
                Le lecteur frame par frame n'est pas fonctionnel en pratique.
Impact      : Fonctionnalité visible dans le code mais non accessible utilisateur.

-----------------------------------------------------------------------
A5. HEATMAP NON IMPLEMENTEE  [écart par déprioritisation : Could]
-----------------------------------------------------------------------
CDC prévu   : Heatmap des positions visitées par chaque agent en Battle Arena
Réalisé     : Non implémenté.
Raison      : Déprioritisé au profit de la robustesse du cœur fonctionnel.
Impact      : Visualisation avancée absente. Pas d'impact sur les fonctionnalités
              principales.

-----------------------------------------------------------------------
A6. EXPORT PDF NON REALISE  [écart par déprioritisation : Could]
-----------------------------------------------------------------------
CDC prévu   : Export PDF des statistiques (jsPDF mentionné dans le CDC)
Réalisé     : Export CSV et JSON disponibles dans StatsComparison.jsx.
              jsPDF n'est pas utilisé dans l'implémentation finale.
Impact      : Mineur. CSV et JSON couvrent les besoins d'export de données.

-----------------------------------------------------------------------
A7. RESPONSIVE MOBILE NON REALISE  [écart par priorité : Should]
-----------------------------------------------------------------------
CDC prévu   : Interface utilisable sur mobile
Réalisé     : Application desktop uniquement. Le canvas et la mise en page
              ne s'adaptent pas aux petits écrans (<768px).
Impact      : Utilisation sur smartphone impossible ou dégradée.

-----------------------------------------------------------------------
A8. DEPLOIEMENT EN LIGNE NON REALISE  [écart délibéré : Could]
-----------------------------------------------------------------------
CDC prévu   : Déploiement public de l'application
Réalisé     : Application locale uniquement (localhost:5174 / localhost:8000)
Raison      : Nécessite Docker + infrastructure cloud hors scope SAE.
Impact      : Démo uniquement sur machine locale.

-----------------------------------------------------------------------
A9. SONS WEB AUDIO API NON IMPLEMENTES  [écart délibéré : Won't]
-----------------------------------------------------------------------
CDC prévu   : Sons (miam nourriture, boom mort, level-up)
Réalisé     : Le hook useSoundEffects.js existe mais les sons ne sont pas
              implémentés (fonctions vides / stub sans logique Audio).
Impact      : Confort UI uniquement, aucun impact fonctionnel.

-----------------------------------------------------------------------
A10. MULTIJOUEUR EN LIGNE  [Won't — hors scope CDC]
-----------------------------------------------------------------------
CDC prévu   : Hors scope (clairement Won't)
Réalisé     : Non implémenté
Impact      : Aucun.

===========================================================================
PARTIE B — ECARTS RAPPORT DE CONCEPTION vs IMPLEMENTATION REELLE
===========================================================================

-----------------------------------------------------------------------
B1. PIVOT FRONTEND-FIRST → BACKEND-FIRST  [écart de processus]
-----------------------------------------------------------------------
Conception  : Architecture planifiée sans ordre de développement défini
Réalisé     : Le développement a démarré frontend-first (React + mocks).
              Pivot vers backend-first face à la difficulté de valider les
              composants React sans backend opérationnel. Les données mockées
              ne correspondaient pas à la structure réelle du backend.
Conséquence : Backend solidement testé (107 tests pytest) avant intégration
              frontend. Léger retard sur les fonctionnalités UI avancées
              (replay, heatmap non finalisées).

-----------------------------------------------------------------------
B2. PROTOCOLE DELTA (game_delta) NON PREVU  [amélioration ajoutée]
-----------------------------------------------------------------------
Conception  : WebSocket envoyant un game_state complet à chaque tick
Réalisé     : Deux types de messages distincts :
              - "game_state" (type) : premier frame complet (snake, food, obstacles,
                gridSize, mode, score, game_over, astar_path, rl_path…)
              - "game_delta" (type) : frames suivantes — uniquement les champs
                dynamiques (snake, food, score, game_over, astar_path ou rl_path)
Raison      : Réduit la bande passante de ~60% par frame. Les champs statiques
              (obstacles statiques, gridSize, mode) ne sont pas retransmis.
Impact côté front : deux réducteurs Redux distincts dans gameSlice.js :
              setGameState (premier frame) et applyDelta (frames suivantes).

-----------------------------------------------------------------------
B3. CANVAS BI-COUCHE NON PREVU  [amélioration ajoutée]
-----------------------------------------------------------------------
Conception  : Un seul canvas pour l'ensemble du rendu de la grille
Réalisé     : Deux canvas superposés (position:absolute) dans Snake.jsx :
              1. bgCanvas     — fond dégradé + lignes de grille + obstacles statiques
                                redessiné SEULEMENT si gridSize/cellSize/mode change
              2. entityCanvas — obstacles dynamiques + nourriture + serpent +
                                overlay chemin A* ou RL + compteur FPS
                                redessiné à CHAQUE tick de jeu
Raison      : ~90% des frames ne redessinentque la couche entités (légère).
              Amélioration significative des performances de rendu.
cellSize    : Math.max(10, Math.floor(Math.min(w, h) / gridSize))
              Calculé depuis les props availableWidth/availableHeight passées
              par GameGrid via ResizeObserver.

-----------------------------------------------------------------------
B4. ORJSON NON MENTIONNE  [optimisation ajoutée]
-----------------------------------------------------------------------
Conception  : Sérialisation JSON standard (json.dumps)
Réalisé     : orjson (v3.10.7) pour sérialisation backend. Messages envoyés
              en bytes via websocket.send_bytes(orjson.dumps(data)).
              Côté frontend : WSService lit les Blob binaires et convertit
              via event.data.text() avant JSON.parse().
Raison      : orjson est 3-5x plus rapide que json standard.
              Impact mesurable sur les frames fréquentes (5-20/s).

-----------------------------------------------------------------------
B5. GAMMA = 0.9 (CONCEPTION DISAIT 0.95)  [écart de paramètre]
-----------------------------------------------------------------------
Conception  : gamma (discount factor) = 0.95 mentionné dans les docs
Réalisé     : RL_GAMMA = 0.9 dans backend/config.py (configurable via env)
Raison      : Valeur ajustée expérimentalement. gamma=0.9 converge plus vite
              sur des épisodes courts car il accorde moins d'importance aux
              récompenses lointaines.
Impact      : Légèrement plus "myope" mais meilleures performances sur 80 épisodes.

-----------------------------------------------------------------------
B6. BACKOFF RECONNEXION : BASE 1000 ms (CONCEPTION DISAIT 100 ms)
-----------------------------------------------------------------------
Conception  : Backoff exponentiel avec 100 ms comme valeur de départ
Réalisé     : BACKOFF_BASE_MS = 1000 ms dans services/websocket.js
              Max = 30 000 ms. Formule : min(30000, 1000 * 2^retryCount)
              Séquence : 1s → 2s → 4s → 8s → 16s → 30s → 30s → ...
Raison      : 100 ms serait trop agressif et pourrait surcharger le serveur
              lors d'un redémarrage long du backend.

-----------------------------------------------------------------------
B7. VITE PROXY POUR /api ET /ws  [non mentionné dans la conception]
-----------------------------------------------------------------------
Conception  : Appels directs frontend (port 5174) → backend (port 8000)
              avec gestion CORS côté FastAPI uniquement
Réalisé     : vite.config.js configure un proxy transparent en dev :
                /api  → http://127.0.0.1:8000  (requêtes REST axios)
                /ws   → ws://127.0.0.1:8000   (connexion WebSocket)
              Ainsi, api.js utilise '/api/...' (sans port) et le proxy
              redirige transparentement vers le backend.
Avantage    : Pas besoin de hardcoder l'URL du backend dans le code frontend.
Note        : FastAPI a quand même CORSMiddleware (ports 5173-5176 autorisés)
              pour les cas hors proxy (tests, production).

-----------------------------------------------------------------------
B8. DEUX ROUTEURS AGENTS DANS MAIN.PY  [détail non prévu]
-----------------------------------------------------------------------
Conception  : Un seul routeur /api/agent pour toutes les opérations agents
Réalisé     : main.py monte deux routeurs distincts liés aux agents :
              - agents_routes.py → préfixe /api/agents (CRUD BDD, lecture)
              - agent_routes.py  → préfixe /api/agent  (init, step, save)
Raison      : Séparation claire : CRUD base de données vs logique de simulation.

-----------------------------------------------------------------------
B9. RESIZEOBSERVER DANS GAMEGRID  [non mentionné dans la conception]
-----------------------------------------------------------------------
Conception  : Taille du canvas calculée depuis window.innerWidth/innerHeight
Réalisé     : GameGrid.jsx utilise un ResizeObserver sur son propre conteneur
              div pour mesurer la largeur/hauteur réellement disponible, puis
              passe availableWidth / availableHeight à Snake.jsx via props.
Raison      : Robustesse : s'adapte si le panneau latéral change de taille
              ou si la fenêtre est redimensionnée, sans écouter window globalement.
Note test   : ResizeObserver absent de jsdom → mock global nécessaire dans setup.js.

-----------------------------------------------------------------------
B10. MOCKS JSDOM REQUIS POUR LES TESTS  [non prévu dans la conception]
-----------------------------------------------------------------------
Conception  : Tests Vitest sans mention des contraintes jsdom
Réalisé     : setup.js doit fournir deux mocks globaux :
              1. global.ResizeObserver — utilisé par Recharts ResponsiveContainer
                 dans TrainingPanel/BattleArena
              2. HTMLCanvasElement.prototype.getContext — retourne un mock ctx
                 couvrant toutes les méthodes 2D (fillRect, arc, roundRect,
                 createLinearGradient, quadraticCurveTo, fillText, save, restore…)
Raison      : jsdom ne simule pas l'API Canvas HTML5 ni ResizeObserver.
              Sans ces mocks, BattleArena.test.jsx et TrainingPanel.test.jsx
              échouent avec "Cannot read properties of null (reading 'fillRect')"
              et "ResizeObserver is not defined".

-----------------------------------------------------------------------
RESUME PARTIE B
-----------------------------------------------------------------------
La conception initiale était correcte sur les grandes lignes :
architecture client-serveur, WebSocket, Redux, SQLAlchemy, canvas, agents IA.

Les écarts B sont principalement des OPTIMISATIONS TECHNIQUES découvertes
pendant l'implémentation (protocole delta, bi-canvas, orjson, proxy Vite)
et des AJUSTEMENTS DE PARAMÈTRES (gamma=0.9, backoff=1000ms).

Aucun écart majeur sur l'architecture fonctionnelle prévue.
