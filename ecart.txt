===========================================================================
FICHIER DES ECARTS — Snake AI SAE4 UPJV 2024/2025
===========================================================================
Ce fichier recense tous les écarts entre :
  A) Le Cahier des Charges (CDC) et le projet livré
  B) Le rapport de conception initiale et l'implémentation réelle
===========================================================================


===========================================================================
PARTIE A — ECARTS CDC vs REALISE
===========================================================================

-----------------------------------------------------------------------
A1. ACCES SANS INSTALLATION / HEBERGEMENT EN LIGNE  [écart : Could]
-----------------------------------------------------------------------
CDC prévu   : F1 — Application accessible sans installation (hébergement web)
Réalisé     : Application locale uniquement (localhost:5174 / localhost:8000)
Raison      : Nécessite Docker + infrastructure cloud (Render, Railway, VPS)
              hors scope SAE. start.bat/start.command automatisent l'installation
              locale en 1 clic.
Impact      : Démo uniquement sur machine locale. L'évaluateur doit avoir
              Python 3.10+ et Node.js 18+ installés.

-----------------------------------------------------------------------
A2. INTERFACE RESPONSIVE MOBILE  [écart : Should]
-----------------------------------------------------------------------
CDC prévu   : F1 — Interface utilisable sur écrans mobiles et tablettes
Réalisé     : Application desktop uniquement. Le canvas et la mise en page
              ne s'adaptent pas aux écrans < 768px.
Raison      : Déprioritisé. La grille 25×25 sur canvas ne se prête pas
              facilement à un affichage mobile sans refonte majeure du layout.
Impact      : Utilisation sur smartphone impossible ou très dégradée.

-----------------------------------------------------------------------
A3. VUE 3 AGENTS SIMULTANES (ThreeGridView)  [écart : Must partiellement]
-----------------------------------------------------------------------
CDC prévu   : F3 — Affichage simultané des 3 agents (Joueur + A* + RL)
              dans une vue à trois grilles côte à côte
Réalisé     : BattleArena affiche 2 agents en simultané (A* et RL).
              La vue 3 agents (Joueur humain + A* + RL) n'est pas implémentée.
              Le composant ThreeGridView.jsx n'existe pas.
              Le JoueurHumain (contrôle clavier) n'est pas implémenté.
Raison      : Complexité UI élevée pour 3 canvas synchronisés + gestion du
              clavier en parallèle du WebSocket. Déprioritisé au profit de
              la robustesse de la Battle Arena 2 agents.
Impact      : L'exigence F3 du CDC est partiellement réalisée.
              BattleArena remplace ThreeGridView avec 2 agents au lieu de 3.

-----------------------------------------------------------------------
A4. COURBE D'APPRENTISSAGE EN TEMPS REEL  [écart : Should]
-----------------------------------------------------------------------
CDC prévu   : F5 — Courbe d'apprentissage affichée en direct pendant
              l'entraînement RL (mise à jour à chaque épisode)
Réalisé     : TrainingPanel affiche le graphique Recharts des épisodes déjà
              enregistrés en BDD. La mise à jour pendant l'entraînement actif
              n'est pas en temps réel (pas de push WebSocket pour les métriques
              RL). L'utilisateur clique "Actualiser" pour voir les nouveaux points.
Impact      : Fonctionnalité présente mais non temps réel comme prévu dans F5.

-----------------------------------------------------------------------
A5. REPLAY FRAME PAR FRAME NON FONCTIONNEL  [écart partiel]
-----------------------------------------------------------------------
CDC prévu   : F6 — Replay des parties enregistrées, lecture frame par frame
              avec contrôles (play/pause/vitesse)
Réalisé     : - La route backend GET /api/stats/replay/{game_id} EXISTE et
                retourne les événements de la partie (table game_events).
              - La table game_events enregistre chaque mouvement (move, food,
                collision) avec timestamp, direction, score.
              - Le composant ReplayViewer.jsx existe dans le code source.
              - MAIS : ReplayViewer.jsx n'est pas connecté à la route backend.
                Le lecteur frame par frame n'est pas fonctionnel en pratique.
Impact      : Fonctionnalité visible dans le code (backend + composant)
              mais non accessible par l'utilisateur.

-----------------------------------------------------------------------
A6. STATISTIQUES AVANCEES MANQUANTES  [écart : Could]
-----------------------------------------------------------------------
CDC prévu   : F8 — Histogrammes de distribution des scores, heatmap des
              positions visitées, export PDF (jsPDF), tableau comparatif
              détaillé avec efficacité, temps de calcul, taux de survie
Réalisé     : - Histogrammes : non implémentés
              - Heatmap : non implémentée
              - Export PDF (jsPDF) : non réalisé
              - Export CSV et JSON disponibles dans StatsComparison.jsx
              - Tableau comparatif : présent mais sans temps de calcul A*
                ni breakdown détaillé par mode (obstacles/normal)
Raison      : Déprioritisé au profit de la robustesse du cœur fonctionnel.
Impact      : Visualisation avancée partielle. CSV/JSON couvrent l'essentiel.

-----------------------------------------------------------------------
A7. SONS WEB AUDIO API NON IMPLEMENTES  [écart : Could]
-----------------------------------------------------------------------
CDC prévu   : F9 — Sons (miam nourriture, boom mort, musique de fond,
              level-up) via Web Audio API
Réalisé     : Le hook useSoundEffects.js existe mais contient des stubs
              vides sans logique Audio réelle.
Impact      : Confort UI uniquement, aucun impact fonctionnel.

-----------------------------------------------------------------------
A8. OBJECTIFS DE PERFORMANCE NON ATTEINTS  [écart de résultat]
-----------------------------------------------------------------------
CDC prévu   : A* score moyen > 150, RL score moyen > 100 (sur 20×20)
Réalisé     : Sur grille 25×25 (plus grande que le CDC) :
              - A* : score moyen 31.79, meilleur 42, survie 98.75%
              - RL  : score moyen 9.10,  meilleur 23, survie 88.75%
              (80 épisodes d'entraînement RL — très faible vs objectif)
Raison      : - Grille 25×25 plus difficile qu'une 20×20 (plus de chemin)
              - RL tabulaire (254 états) vs DQN nécessaire pour 100+
              - Seulement 80 épisodes d'entraînement vs milliers recommandés
Impact      : Les objectifs chiffrés du CDC ne sont pas atteints.
              Les scores A* restent cohérents avec un algorithme optimal.
              Les scores RL sont cohérents avec un Q-Learning peu entraîné.

-----------------------------------------------------------------------
A9. SOCKET.IO → WEBSOCKET NATIF FASTAPI  [écart intentionnel]
-----------------------------------------------------------------------
CDC prévu   : Communication temps réel via Socket.IO
Réalisé     : WebSocket natif FastAPI (Starlette), sérialisation orjson bytes
Raison      : - FastAPI intègre nativement WebSocket sans dépendance externe
              - Socket.IO ajoute ~80 Ko au bundle frontend
              - orjson 3-5x plus rapide que JSON standard
              - Reconnexion automatique réimplémentée dans WSService (backoff expo)
Impact      : Aucun impact fonctionnel. WebSocket natif équivaut à Socket.IO
              pour ce cas d'usage.

-----------------------------------------------------------------------
A10. CHART.JS → RECHARTS  [écart de bibliothèque]
-----------------------------------------------------------------------
CDC prévu   : Visualisation avec Chart.js
Réalisé     : Recharts (composants React natifs, pas de ref canvas manuel)
Raison      : Recharts s'intègre mieux avec React/Redux sans manipulation
              impérative du DOM canvas.
Impact      : Aucun impact fonctionnel. API différente mais résultat équivalent.

-----------------------------------------------------------------------
A11. JEST → VITEST  [écart de framework de test]
-----------------------------------------------------------------------
CDC prévu   : Tests frontend avec Jest
Réalisé     : Vitest (compatible Vite, même API que Jest, plus rapide)
Raison      : Vitest est le standard de facto pour les projets Vite/React.
              Configuration identique à Jest, migration transparente.
Impact      : Aucun impact. Vitest produit les mêmes rapports que Jest.

-----------------------------------------------------------------------
A12. DQN (réseau de neurones) NON REALISE  [écart : Could]
-----------------------------------------------------------------------
CDC prévu   : Possibilité d'implémenter un Deep Q-Network (DQN) comme
              extension avancée
Réalisé     : Q-Learning tabulaire uniquement (Q-table JSON, 254 états visités)
Raison      : DQN nécessite PyTorch + GPU recommandé. Complexité très élevée
              hors scope d'une SAE. La Q-table tabulaire est suffisante pour
              la démonstration et reste explicable (O(1), 11 features lisibles).
Impact      : Score Q-Learning plafonné (~23 max, ~9 moyen sur 80 épisodes).
              DQN permettrait d'atteindre 50-100+ avec assez d'entraînement.

-----------------------------------------------------------------------
RESUME PARTIE A
-----------------------------------------------------------------------
Exigences Must réalisées   : Jeu fonctionnel, A*, RL (tabulaire), interface,
                             BDD, CI/CD GitHub Actions — TOUTES réalisées
Exigences Should réalisées : Overlay chemin A*/RL, obstacles dynamiques,
                             mode performance, export CSV/JSON — partiellement
Exigences Could non faites : DQN, replay frontend, heatmap, PDF, sons,
                             hébergement en ligne
Ecarts majeurs             : Vue 3 agents (F3), replay non connecté (F6),
                             scores cibles non atteints (A8)


===========================================================================
PARTIE B — ECARTS RAPPORT DE CONCEPTION vs IMPLEMENTATION REELLE
===========================================================================

-----------------------------------------------------------------------
B1. THREEGRIDVIEW NON IMPLEMENTEE  [composant non réalisé]
-----------------------------------------------------------------------
Conception  : ThreeGridView — composant React affichant 3 canvas côte à côte
              (Joueur, A*, RL) avec synchronisation des ticks via un seul
              WebSocket multiplexé
Réalisé     : BattleArena.jsx affiche 2 canvas (A* et RL) via deux WebSockets
              indépendants. Pas de ThreeGridView. Pas de JoueurHumain.
Raison      : Synchronisation 3 canvas + clavier en parallèle WebSocket trop
              complexe. La BattleArena 2 agents répond à l'essentiel du besoin.

-----------------------------------------------------------------------
B2. GESTIONNAIRE WEBSOCKET MULTI-CONNEXION NON PREVU  [architecture simplifiée]
-----------------------------------------------------------------------
Conception  : GestionnaireWebSocket — classe unique gérant N connexions
              WebSocket simultanées avec un pool de connexions
Réalisé     : WSService (services/websocket.js) gère une seule connexion
              à la fois. BattleArena instancie deux WSService indépendants
              (un par agent). Pas de pool centralisé.
Raison      : Deux instances indépendantes suffisent. Le pool centralisé
              apportait de la complexité sans gain fonctionnel pour 2 agents.

-----------------------------------------------------------------------
B3. JOUEUR HUMAIN NON IMPLEMENTE  [classe non réalisée]
-----------------------------------------------------------------------
Conception  : JoueurHumain — classe Python héritant de AgentBase,
              recevant les commandes clavier du frontend via WebSocket
              et les transmettant au moteur de jeu
Réalisé     : Aucune classe JoueurHumain. Le mode joueur humain n'existe pas.
              Les agents disponibles sont : A* (AgentAstar), RL (AgentRL),
              Aléatoire (AgentAleatoire).
Impact      : F3 du CDC (vue 3 agents avec joueur humain) non réalisée.

-----------------------------------------------------------------------
B4. AGENTRL.ENTRAINER() DEPLACE DANS TRAINER  [refactoring d'architecture]
-----------------------------------------------------------------------
Conception  : AgentRL.entrainer(n_episodes) — méthode intégrée dans la
              classe AgentRL, appelable directement
Réalisé     : La logique d'entraînement est dans RLTrainer (rl_trainer.py).
              AgentRL gère uniquement la politique (choisir_action, mettre_a_jour).
              RLTrainer orchestre les épisodes, la BDD, les callbacks de progression.
Raison      : Séparation des responsabilités : politique ≠ orchestration.
              Permet de tester AgentRL indépendamment de la BDD.

-----------------------------------------------------------------------
B5. CHART.JS → RECHARTS  [changement de bibliothèque frontend]
-----------------------------------------------------------------------
Conception  : Chart.js pour tous les graphiques (courbe d'apprentissage,
              comparaison, histogrammes)
Réalisé     : Recharts — composants React déclaratifs (LineChart, BarChart,
              RadarChart). Pas de Chart.js dans le projet.
Raison      : Recharts s'intègre naturellement avec React sans manipulation
              impérative de canvas. Meilleure cohérence avec l'architecture Redux.

-----------------------------------------------------------------------
B6. HOWLER.JS NON UTILISE  [bibliothèque abandonnée]
-----------------------------------------------------------------------
Conception  : Howler.js pour la gestion des sons (chargement, lecture,
              volume, cache audio)
Réalisé     : useSoundEffects.js contient des stubs vides. Howler.js n'est
              pas dans package.json.
Raison      : Sons déprioritisés (priorité Could). Howler.js non installé.

-----------------------------------------------------------------------
B7. TABLE AGENTS : CHAMP DESCRIPTION MANQUANT  [écart de schéma BDD]
-----------------------------------------------------------------------
Conception  : Table agents avec champs : id, nom, type, description,
              parametres (JSON), date_creation, actif
Réalisé     : Table agents sans champ `description`.
              Champs réels : id, name, type, parameters (JSON), created_at,
              is_active, best_score, total_games, avg_score.
              Ajout de best_score, total_games, avg_score non prévus en
              conception (dénormalisation pour performance).

-----------------------------------------------------------------------
B8. AGENT_STATS : STRUCTURE FLEXIBLE → COLONNES FIXES  [écart de schéma]
-----------------------------------------------------------------------
Conception  : Table agent_statistics avec structure flexible (JSON) pour
              stocker des métriques variables selon le type d'agent
Réalisé     : Table agent_stats avec colonnes fixes :
              avg_score, max_score, min_score, total_games,
              avg_survival_rate, avg_path_length, obstacles_mode (bool)
Raison      : Colonnes fixes permettent des requêtes SQL directes et des
              index. La structure JSON flexible est plus difficile à requêter.

-----------------------------------------------------------------------
B9. ETATJEU : ETATS PRET / TERMINE NON IMPLEMENTE  [énumération simplifiée]
-----------------------------------------------------------------------
Conception  : EtatJeu énumération avec états : PRET, EN_COURS, EN_PAUSE,
              TERMINE, ERREUR
Réalisé     : États gérés par des booléens dans le GameEngine :
              is_running (bool), game_over (bool). Pas d'enum EtatJeu.
              Le frontend gère les états via Redux (wsSlice + gameSlice).
Raison      : Deux booléens suffisent pour les transitions nécessaires.
              L'enum formelle aurait été over-engineering.

-----------------------------------------------------------------------
B10. GAMMA = 0.9 (CONCEPTION DISAIT 0.95)  [écart de paramètre]
-----------------------------------------------------------------------
Conception  : gamma (discount factor) = 0.95
Réalisé     : RL_GAMMA = 0.9 dans backend/config.py (configurable via env)
Raison      : gamma=0.9 converge plus vite sur des épisodes courts car il
              accorde moins d'importance aux récompenses lointaines.
Impact      : Légèrement plus "myope" mais meilleures performances sur 80 épisodes.

-----------------------------------------------------------------------
B11. JEST → VITEST  [framework de test]
-----------------------------------------------------------------------
Conception  : Tests frontend avec Jest + React Testing Library
Réalisé     : Vitest (compatible Vite, même API que Jest).
              vitest.config.js hérite de vite.config.js, environnement jsdom.
Raison      : Vitest est standard pour les projets Vite. Pas de configuration
              Babel supplémentaire requise.

-----------------------------------------------------------------------
B12. NUMPY VECTORISE POUR A* NON REALISE  [optimisation non faite]
-----------------------------------------------------------------------
Conception  : Utilisation de NumPy pour représenter la grille et accélérer
              les calculs de voisins dans A*
Réalisé     : A* utilise des listes Python standard (set pour closed_set,
              heapq pour open_set, list de tuples pour la grille).
Raison      : Les performances sont suffisantes sur une grille 25×25.
              NumPy est utilisé uniquement dans AgentRL (features numpy array).
Impact      : Aucun impact perceptible. A* reste O(n log n) sur 625 cellules.

-----------------------------------------------------------------------
B13. BOUCLE 60 FPS requestAnimationFrame NON IMPLEMENTEE  [écart rendu]
-----------------------------------------------------------------------
Conception  : Boucle de rendu frontend à 60 FPS via requestAnimationFrame
              indépendante du tick serveur
Réalisé     : Rendu déclenché par chaque message WebSocket reçu (event-driven).
              Pas de boucle requestAnimationFrame autonome.
              Fréquence réelle : 5-20 FPS selon la vitesse de jeu configurée.
Raison      : Le rendu event-driven évite de dessiner des frames inutiles
              (rien ne change entre deux ticks serveur).

-----------------------------------------------------------------------
B14. PROTOCOLE DELTA (game_delta) NON PREVU  [amélioration ajoutée]
-----------------------------------------------------------------------
Conception  : WebSocket envoyant un game_state complet à chaque tick
Réalisé     : Deux types de messages :
              - "game_state" : premier frame complet (snake, food, obstacles,
                gridSize, mode, score, game_over, astar_path, rl_path)
              - "game_delta" : frames suivantes — uniquement les champs
                dynamiques (snake, food, score, game_over, astar_path ou rl_path)
Raison      : Réduit la bande passante de ~60% par frame.
Impact      : Deux réducteurs Redux distincts dans gameSlice.js :
              setGameState (premier frame) et applyDelta (frames suivantes).

-----------------------------------------------------------------------
B15. CANVAS BI-COUCHE NON PREVU  [amélioration de performance ajoutée]
-----------------------------------------------------------------------
Conception  : Un seul canvas pour l'ensemble du rendu
Réalisé     : Deux canvas superposés (position:absolute) dans Snake.jsx :
              1. bgCanvas     — fond + grille + obstacles statiques
                                redessiné SEULEMENT si gridSize/cellSize/mode change
              2. entityCanvas — serpent + nourriture + overlay chemin + FPS
                                redessiné à chaque tick WebSocket
Raison      : ~90% des frames ne redessinentque la couche entités.
              Amélioration significative des performances de rendu.

-----------------------------------------------------------------------
B16. ORJSON NON MENTIONNE  [optimisation ajoutée]
-----------------------------------------------------------------------
Conception  : Sérialisation JSON standard (json.dumps / json.loads)
Réalisé     : orjson (v3.10.7) côté backend. Messages envoyés en bytes :
              websocket.send_bytes(orjson.dumps(data))
              Frontend : WSService lit les Blob binaires via event.data.text()
              avant JSON.parse().
Raison      : orjson est 3-5x plus rapide que json standard.

-----------------------------------------------------------------------
B17. VITE PROXY NON MENTIONNE  [infrastructure dev ajoutée]
-----------------------------------------------------------------------
Conception  : Appels directs frontend → backend avec CORS FastAPI
Réalisé     : vite.config.js proxy transparent :
                /api → http://127.0.0.1:8000
                /ws  → ws://127.0.0.1:8000
              api.js utilise '/api/...' sans port hardcodé.
Note        : FastAPI conserve CORSMiddleware pour les cas hors proxy.

-----------------------------------------------------------------------
B18. PIVOT FRONTEND-FIRST → BACKEND-FIRST  [écart de processus]
-----------------------------------------------------------------------
Conception  : Ordre de développement non défini
Réalisé     : Démarrage frontend-first (React + mocks). Pivot vers backend-first
              face à la difficulté de valider les composants React sans backend
              opérationnel (données mockées ≠ structure réelle).
Conséquence : Backend solidement testé (107 tests pytest) avant intégration
              frontend. Léger retard sur les fonctionnalités UI avancées.

-----------------------------------------------------------------------
RESUME PARTIE B
-----------------------------------------------------------------------
La conception était correcte sur les grandes lignes :
architecture client-serveur, WebSocket, Redux, SQLAlchemy, canvas, agents IA.

Ecarts principaux :
- Composants non réalisés  : ThreeGridView, JoueurHumain, GestionnaireWebSocket pool
- Refactoring              : AgentRL.entrainer() → RLTrainer, EtatJeu → booléens
- Bibliothèques changées   : Chart.js→Recharts, Howler.js absent, Jest→Vitest
- Schéma BDD               : champ description absent, structure flexible→colonnes fixes
- Paramètres               : gamma 0.95→0.9
- Améliorations non prévues : protocole delta, bi-canvas, orjson, proxy Vite

Aucun écart majeur sur l'architecture fonctionnelle globale.
